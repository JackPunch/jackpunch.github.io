HTTP（HyperText Transfer Protocol，超文本传输协议）是TCP/IP协议簇中的**应用层协议**，是万维网（WWW）数据通信的核心基础，定义了客户端（浏览器、APP等）与Web服务器之间的通信格式、交互规则与数据传输规范，默认使用80端口。

它位于OSI七层模型的第7层（应用层）、TCP/IP四层模型的应用层，底层默认基于可靠的TCP协议实现传输（HTTP/3例外，基于UDP的QUIC协议），是目前互联网中使用最广泛的应用层协议。

---

## 一、HTTP协议的发展历程与版本演进
HTTP从诞生至今经历了5个核心版本，核心演进方向是**提升传输性能、增强安全性、扩展功能边界**。

| 版本 | 发布时间 | 核心特性与里程碑 |
|------|----------|------------------|
| HTTP/0.9 | 1991年 | 原型极简版本，仅支持`GET`请求方法；无请求头、响应头、状态码；仅支持传输纯HTML文本；请求完成立即关闭TCP连接（短连接） |
| HTTP/1.0 | 1996年（RFC 1945） | 首个正式标准化版本；新增`POST`/`HEAD`方法；引入请求头、响应头、状态码；通过`Content-Type`支持图片、视频、二进制等多种数据格式；默认短连接，每次请求需新建TCP连接 |
| HTTP/1.1 | 1997年（RFC 2616，2014年修订RFC 7230-7235） | 目前全球最广泛使用的版本；默认开启**长连接（keep-alive）**，TCP连接可复用；新增`PUT`/`DELETE`/`OPTIONS`等方法；强制要求`Host`字段，支持单服务器多虚拟主机；新增缓存控制、断点续传、身份认证机制；支持管道化请求（未彻底解决队头阻塞） |
| HTTP/2 | 2015年（RFC 7540） | 二进制分帧协议，性能大幅提升；核心实现**多路复用**，单TCP连接可并行传输多个请求/响应，彻底解决应用层队头阻塞；支持`HPACK`头部压缩、服务器主动推送、流优先级控制 |
| HTTP/3 | 2022年正式标准化（RFC 9114） | 底层重构，放弃TCP，基于UDP的**QUIC协议**实现传输；彻底解决传输层队头阻塞；0-RTT/1-RTT握手大幅降低连接延迟；支持连接迁移（网络切换无需重建连接）；内置TLS 1.3，默认加密 |

## 二、HTTP协议的核心特性
### 1. 请求-响应模型
通信只能由**客户端主动发起**，服务器被动接收并响应请求，无法主动向客户端推送数据（HTTP/2的服务器推送是预推送资源，并非主动发起双向通信）。

### 2. 无状态性
协议本身不保存客户端的任何状态信息，每次请求都是完全独立的，服务器不会记住之前的请求上下文。
- 优点：简化协议设计，降低服务器存储与负载压力
- 缺点：无法支持需要用户状态的场景（如登录、购物车），因此衍生出`Cookie/Session`机制解决状态保持问题

### 3. 灵活可扩展
- 可自定义请求方法、头部字段，按需扩展功能
- 通过`Content-Type`头部支持任意格式数据传输（JSON、XML、音视频、文件流等），不局限于超文本

### 4. 可靠传输
HTTP/1.x、HTTP/2基于TCP协议，依托TCP的三次握手、超时重传、拥塞控制机制保证数据完整有序交付；HTTP/3基于QUIC协议，也在UDP之上实现了可靠传输能力。

### 5. 原生明文传输
HTTP报文以明文形式传输，不做任何加密处理，存在三大核心安全风险：数据窃听、内容篡改、身份冒充，这也是HTTPS诞生的核心原因。

## 三、HTTP协议完整工作流程
以最常用的HTTP/1.1为例，一次完整的HTTP请求-响应全流程如下：
1. **DNS域名解析**：客户端将请求的域名（如www.baidu.com）通过DNS服务器，解析为服务器对应的IP地址
2. **TCP连接建立**：客户端与服务器IP的80端口（HTTPS为443端口）完成TCP三次握手，建立可靠的TCP连接
3. **发送HTTP请求报文**：客户端通过TCP连接，向服务器发送符合HTTP规范的请求报文
4. **服务器处理请求**：服务器接收并解析请求报文，根据请求方法、URL路径，读取静态资源/调用后端服务，生成响应报文
5. **返回HTTP响应报文**：服务器将生成的响应报文，通过TCP连接回传给客户端
6. **连接管理**：若报文设置`Connection: close`，则通过TCP四次挥手关闭连接；若为`keep-alive`，则保持连接，复用处理后续请求
7. **客户端处理响应**：客户端接收并解析响应内容，完成HTML渲染、JSON数据解析、资源下载等操作，结束本次交互

## 四、HTTP报文核心结构
HTTP报文分为**请求报文（客户端→服务器）**和**响应报文（服务器→客户端）**两类，结构高度统一，均由4个核心部分组成。

### 1. HTTP请求报文
请求报文由**请求行、请求头、空行、请求体**四部分组成，示例如下：
```http
# 1. 请求行
GET /user/info?uid=123 HTTP/1.1
# 2. 请求头
Host: www.example.com
User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Chrome/120.0.0.0
Accept: application/json
Cookie: token=abcdef123456
Content-Type: application/json
Content-Length: 32
# 3. 空行（必须存在，分隔请求头与请求体）

# 4. 请求体
{"username":"test","age":20}
```

#### （1）请求行
报文第一行，是请求的核心信息，格式为：`请求方法  URL  HTTP版本号`，三个元素用空格分隔。
- 请求方法：定义对资源的操作类型（如GET/POST）
- URL：统一资源定位符，标识要访问的目标资源路径
- 版本号：客户端使用的HTTP协议版本（如HTTP/1.1）

#### （2）请求头（Request Headers）
请求行之后的多行键值对，格式为`字段名: 值`，用于传递客户端的环境、配置、认证、数据格式等信息，核心常用字段如下：
| 字段名 | 核心作用 |
|--------|----------|
| Host | 目标服务器的域名+端口，HTTP/1.1强制必填，支持虚拟主机 |
| User-Agent | 客户端标识（浏览器、操作系统、设备信息） |
| Accept | 客户端可接收的响应数据类型（如text/html、application/json） |
| Accept-Encoding | 客户端支持的压缩格式（如gzip、br） |
| Cookie | 客户端向服务器传递的状态信息，自动携带 |
| Authorization | 身份认证信息（如Token、Basic Auth） |
| Content-Type | 请求体的数据格式（如application/json、multipart/form-data） |
| Content-Length | 请求体的字节长度 |
| Connection | 连接管理（keep-alive/close） |

#### （3）空行
请求头与请求体之间必须有一个单独的空行（CRLF换行符），作用是标识请求头结束，请求体开始，是HTTP报文格式的强制要求。

#### （4）请求体（Request Body）
可选部分，用于传递请求的业务数据。GET方法规范不推荐使用请求体，POST/PUT等方法的核心数据均放在请求体中，支持任意格式的数据。

### 2. HTTP响应报文
响应报文由**状态行、响应头、空行、响应体**四部分组成，示例如下：
```http
# 1. 状态行
HTTP/1.1 200 OK
# 2. 响应头
Server: Nginx
Content-Type: application/json; charset=utf-8
Content-Length: 56
Set-Cookie: token=abcdef123456; HttpOnly; Secure
Cache-Control: max-age=3600
# 3. 空行

# 4. 响应体
{"code":0,"msg":"success","data":{"username":"test","age":20}}
```

#### （1）状态行
报文第一行，格式为：`HTTP版本号  状态码  状态描述短语`，三个元素用空格分隔。
- 版本号：服务器使用的HTTP协议版本
- 状态码：3位数字，标识请求的处理结果（如200/404/500）
- 状态描述：对状态码的文本说明（如OK、Not Found）

#### （2）响应头（Response Headers）
状态行之后的多行键值对，格式为`字段名: 值`，用于传递服务器的配置、资源信息、缓存策略、状态设置等，核心常用字段如下：
| 字段名 | 核心作用 |
|--------|----------|
| Server | 服务器软件标识（如Nginx、Apache） |
| Content-Type | 响应体的数据格式与编码（如text/html; charset=utf-8） |
| Content-Length | 响应体的字节长度 |
| Content-Encoding | 响应体的压缩格式 |
| Set-Cookie | 服务器向客户端设置Cookie，用于状态保持 |
| Location | 重定向目标URL，配合3xx状态码使用 |
| Cache-Control | 缓存控制策略 |
| ETag/Last-Modified | 协商缓存标识 |

#### （3）空行
响应头与响应体之间的强制空行，标识响应头结束。

#### （4）响应体（Response Body）
服务器返回的核心数据，是客户端请求的目标内容，如HTML文档、JSON数据、图片二进制流、音视频文件等。绝大多数响应都包含响应体，仅HEAD方法、304等特殊场景无响应体。

## 五、HTTP标准请求方法详解
基于RFC 7231标准，HTTP定义了8种核心请求方法，用于定义对目标资源的不同操作类型，核心特性如下：

### 1. 核心常用方法
| 方法 | 核心用途 | 幂等性 | 可缓存性 | 核心特点 |
|------|----------|--------|----------|----------|
| GET | 从服务器获取/查询指定资源，不修改数据 | 是 | 是 | 参数拼接在URL的query中，无请求体（规范不禁止但不推荐），受URL长度限制，浏览器默认缓存 |
| POST | 向服务器提交数据，创建/修改资源 | 否 | 仅手动设置缓存头才可缓存 | 数据放在请求体中，无长度限制，支持任意格式数据，常用于表单提交、接口数据上报 |
| PUT | 全量更新服务器指定资源，不存在则创建 | 是 | 否 | 完整覆盖目标资源，多次执行结果一致，用于确定性的更新操作 |
| DELETE | 请求服务器删除指定URI的资源 | 是 | 否 | 多次删除结果一致，用于资源删除操作 |
| OPTIONS | 获取目标资源支持的请求方法，以及服务器跨域配置 | 是 | 否 | 浏览器CORS跨域预检的默认请求方法，无需携带业务数据 |

### 2. 辅助方法
- **HEAD**：与GET逻辑完全一致，但服务器仅返回响应头，不返回响应体。用于获取资源元信息（如文件大小、是否存在），无需下载完整资源，节省带宽。
- **TRACE**：回显服务器收到的请求内容，用于测试、诊断请求链路，生产环境通常禁用（易引发XST跨站追踪攻击）。
- **CONNECT**：用于与目标服务器建立TCP隧道，通常用于HTTPS代理场景，代理服务器仅转发加密数据，不解析内容。

### 补充：GET与POST的核心区别
| 对比维度 | GET | POST |
|----------|-----|------|
| 核心用途 | 资源查询，无副作用 | 数据提交，修改服务器资源 |
| 幂等性 | 幂等，多次执行结果一致 | 非幂等，多次提交可能创建多条数据 |
| 数据位置 | URL的query字符串中 | 请求体中 |
| 数据长度 | 受浏览器/服务器URL长度限制（通常2KB-8KB） | 无硬性限制，可通过服务器配置调整 |
| 缓存 | 浏览器默认缓存 | 默认不缓存 |
| 书签/历史 | 可收藏为书签，保留在浏览器历史 | 不可收藏，不保留在历史 |
| 刷新/回退 | 无害，无二次提交 | 会重新提交数据，浏览器弹出风险提示 |
| 安全性 | 参数明文在URL中，易被记录、抓包 | 数据在请求体中，仅相对隐蔽，HTTP下仍为明文，HTTPS才具备安全性 |

## 六、HTTP状态码详解
HTTP状态码是3位数字，用于标识服务器对请求的处理结果，第一位数字定义状态码的类别，共分为5大类，每类有明确的语义规范。

### 1. 1xx 信息性状态码
临时响应，标识服务器已接收请求，正在处理，需客户端继续操作。
- **100 Continue**：服务器已接收请求头，客户端可继续发送请求体，用于大文件上传场景
- **101 Switching Protocols**：服务器已同意客户端的协议切换请求（如WebSocket升级）

### 2. 2xx 成功状态码
请求已被服务器成功接收、理解并处理。
- **200 OK**：请求成功，响应体包含请求的目标资源（最常用）
- **201 Created**：资源创建成功，常用于POST/PUT请求，响应头包含新资源的URI
- **204 No Content**：请求成功，但无响应体返回，常用于删除、更新操作
- **206 Partial Content**：服务器成功处理了部分GET请求，用于断点续传、分块下载

### 3. 3xx 重定向状态码
需要客户端执行额外操作（通常是跳转）才能完成请求。
- **301 Moved Permanently**：永久重定向，资源URI已永久变更，浏览器会缓存重定向地址，SEO传递权重
- **302 Found**：临时重定向，资源URI临时变更，客户端应继续使用原URI，不缓存
- **304 Not Modified**：协商缓存命中，资源未更新，客户端直接使用本地缓存，无响应体
- **307 Temporary Redirect**：临时重定向，与302的区别是严格禁止请求方法从POST改为GET
- **308 Permanent Redirect**：永久重定向，与301的区别是严格禁止请求方法从POST改为GET

### 4. 4xx 客户端错误状态码
请求存在错误，服务器无法处理，责任在客户端。
- **400 Bad Request**：请求参数、格式错误，服务器无法解析
- **401 Unauthorized**：未认证，请求需要用户身份验证，需携带认证信息重新请求
- **403 Forbidden**：服务器已理解请求，但拒绝执行，通常是权限不足
- **404 Not Found**：请求的资源不存在，URI无效或资源已删除（最常见客户端错误）
- **405 Method Not Allowed**：请求方法不被目标资源支持
- **408 Request Timeout**：服务器等待客户端请求超时
- **413 Payload Too Large**：请求体超过服务器处理限制
- **414 URI Too Long**：请求URL超过服务器长度限制

### 5. 5xx 服务端错误状态码
服务器处理请求时发生内部错误，责任在服务端。
- **500 Internal Server Error**：服务器内部未知错误（最常见服务端错误）
- **502 Bad Gateway**：网关/代理服务器从上游服务器收到无效响应
- **503 Service Unavailable**：服务器暂时不可用，通常是过载、停机维护
- **504 Gateway Timeout**：网关/代理服务器请求上游服务器超时
- **505 HTTP Version Not Supported**：服务器不支持客户端使用的HTTP版本

## 七、HTTP核心机制
### 1. HTTP缓存机制
缓存的核心目标是减少重复请求、降低服务器负载、提升页面加载速度，分为**强缓存**和**协商缓存**两大类。

#### （1）强缓存
不向服务器发送任何请求，直接从客户端本地缓存读取资源，命中时返回`200 OK (from disk cache/memory cache)`。
- **Expires**：HTTP/1.0字段，值为绝对时间，客户端本地时间超过该时间则缓存失效，缺点是依赖客户端本地时间。
- **Cache-Control**：HTTP/1.1字段，优先级高于Expires，核心常用值：
  - `max-age=xxx`：缓存最大有效时间，单位为秒，相对时间，不受本地时间影响
  - `no-cache`：禁用强缓存，强制发起协商缓存
  - `no-store`：完全禁用缓存，每次都请求完整资源
  - `public/private`：控制缓存是否可被代理服务器/CDN缓存

#### （2）协商缓存
强缓存失效后，客户端携带缓存标识向服务器发起请求，服务器根据标识判断资源是否更新：
- 资源未更新：返回`304 Not Modified`，客户端使用本地缓存
- 资源已更新：返回`200 OK`，附带新的资源和新的缓存标识

两组核心缓存标识：
1. **Last-Modified / If-Modified-Since**
   - `Last-Modified`：响应头，服务器返回资源的最后修改时间
   - `If-Modified-Since`：请求头，客户端再次请求时携带该时间，服务器对比资源修改时间判断是否失效
   - 缺点：秒级精度限制，文件内容不变但修改时间变更会误判失效

2. **ETag / If-None-Match**
   - `ETag`：响应头，服务器根据资源内容生成的唯一哈希值，内容不变则ETag不变，优先级高于Last-Modified
   - `If-None-Match`：请求头，客户端再次请求时携带ETag，服务器对比判断资源是否更新
   - 解决了Last-Modified的精度缺陷，判断更精准

### 2. 状态保持机制：Cookie与Session
HTTP无状态的特性，无法记录用户身份，Cookie与Session是解决该问题的核心方案。

#### （1）Cookie
Cookie是存储在客户端浏览器的小型文本数据，单条大小限制约4KB，由服务器通过`Set-Cookie`响应头设置，后续请求自动携带。
- 核心工作流程：
  1. 客户端首次请求服务器，服务器通过`Set-Cookie`向客户端写入Cookie
  2. 浏览器保存Cookie，后续每次请求该域名，都会自动在`Cookie`请求头中携带
  3. 服务器读取Cookie，识别用户身份，保持会话状态
- 核心安全属性：
  - `HttpOnly`：禁止JavaScript读取Cookie，防止XSS攻击窃取
  - `Secure`：仅在HTTPS请求中携带Cookie
  - `SameSite`：控制跨站请求是否携带Cookie，防止CSRF攻击
  - `Domain/Path`：限制Cookie的生效域名与路径

#### （2）Session
Session是存储在服务器端的会话数据，每个会话对应唯一的`Session ID`，用于保存用户的状态信息，安全性远高于Cookie。
- 核心工作流程：
  1. 客户端首次请求，服务器创建唯一的Session ID，同时在服务端生成对应的Session对象，存储用户数据
  2. 服务器将Session ID通过`Set-Cookie`返回给客户端
  3. 客户端后续请求携带包含Session ID的Cookie，服务器读取ID找到对应的Session，识别用户状态
- 特点：数据存储在服务端，无大小限制，安全性高；默认依赖Cookie实现，客户端禁用Cookie时可通过URL重写传递Session ID。

## 八、HTTPS：HTTP的安全升级
HTTPS（HyperText Transfer Protocol Secure）= **HTTP + TLS/SSL**，默认端口443，在HTTP与TCP之间增加了TLS/SSL加密层，是目前互联网的主流协议。

### 1. 解决HTTP的三大核心安全风险
- **窃听风险**：HTTP明文传输，数据可被中间人抓取；HTTPS通过对称加密+非对称加密实现数据加密传输，无法被窃听
- **篡改风险**：HTTP数据可被中间人随意修改；HTTPS通过摘要算法保证数据完整性，篡改可被即时识别
- **冒充风险**：HTTP无法验证服务器身份；HTTPS通过CA机构颁发的数字证书，验证服务器身份，防止钓鱼网站冒充

### 2. 核心工作流程
1. TCP三次握手建立连接
2. TLS握手（TLS 1.3为1-RTT，TLS 1.2为2-RTT）：协商加密套件、验证服务器数字证书、交换会话密钥
3. 加密传输：所有HTTP报文通过会话密钥对称加密后传输
4. 断开连接：TLS关闭通知 + TCP四次挥手

### 3. HTTP与HTTPS核心对比
| 对比维度 | HTTP | HTTPS |
|----------|------|-------|
| 默认端口 | 80 | 443 |
| 安全性 | 明文传输，无加密、认证、完整性校验 | 基于TLS加密，具备完整的安全能力 |
| 性能 | 无加解密开销，连接建立快 | 有TLS握手与加解密开销，HTTP/2/3优化后性能接近甚至反超 |
| SEO | 无权重加成 | 搜索引擎优先收录，SEO权重更高 |
| 证书 | 无需证书 | 需要CA机构颁发的SSL/TLS证书 |
| 协议层级 | 应用层，直接基于TCP | 应用层HTTP + 传输层TLS加密层，基于TCP |

## 九、常见误区与补充说明
1. **误区：POST比GET安全**
   纠正：HTTP原生均为明文传输，POST的参数仅在请求体中相对隐蔽，抓包仍可完整获取，只有HTTPS加密后才具备真正的安全性。

2. **误区：GET请求不能携带请求体**
   纠正：HTTP规范并未禁止GET请求携带请求体，只是绝大多数服务器、浏览器、代理服务器不处理GET的请求体，生产环境强烈不推荐使用。

3. **误区：HTTP/1.1管道化解决了队头阻塞**
   纠正：管道化仅允许并行发送请求，但响应必须按请求顺序返回，前一个响应阻塞会导致后续所有响应阻塞，未解决队头阻塞；HTTP/2多路复用解决了应用层队头阻塞，HTTP/3彻底解决了传输层队头阻塞。

4. **HTTP核心扩展**
   - **WebSocket**：基于HTTP升级的全双工通信协议，解决HTTP无法服务器主动推送的问题，用于实时聊天、直播弹幕、实时数据看板等场景
   - **CORS跨域资源共享**：基于HTTP头部的机制，解决浏览器同源策略限制，允许跨域请求
   - **断点续传**：通过`Range`请求头与`206`状态码实现，大文件分块下载，中断后可继续下载剩余部分，无需重新下载完整文件
